    1. 添加偏移值

      遍历到nums[i]时候，将对应位置nums[i]-1处的元素加上一个大于n的偏移值如+(n+1)，这样位置nums[i]-1处的元素值/(n+1)的结果就是元素值nums[i]目前为止出现的次数。而遍历到位置i时候通过nums[i] % (n+1)就可保证获取到原数组中的值了。

      这种方法可以记录元素值出现的具体次数，比较通用。

    2. 正负标识

      遍历到nums[i]时候，将对应位置nums[i]-1处的元素进行* -1，这样位置nums[i]-1处的元素值为负的话就表明元素值nums[i]目前为止出现过。而遍历到位置i时候通过abs(nums[i])就可以保证获取到原数组中的值了。

      这种方法只能记录元素值出现次数的奇偶性，可以用于判断元素是否出现过等。

Chapter 3: Page Tables

页表是操作系统为每个进程提供自己的私有地址空间和内存的机制。

3.1 分页硬件

RISC-V 页表硬件通过将每个虚拟地址映射到物理地址来连接这两种地址。

xv6 在 Sv39 RISC-V 上运行，因此64位虚拟地址中仅由低39位有被用到。逻辑上，一个RISC-V的页表是一个含有$2^{27}$个页表项(PTE)的数组。每个 PTE 包含一个 44 位物理页号 (PPN) 和一些标志位。分页硬件通过使用虚拟地址39 位中的前 27 位索引到页表中找到 PTE，然后将PTE中的PPN(44位)和原始虚拟地址中剩下的位(12位)合并起来制作一个 56 位物理地址，其前 44 位来自 PTE 中的 PPN，其底部从原始虚拟地址复制 12 位。页表使操作系统能够以 4096 ($2^{12}$) 字节的对齐块的粒度控制虚拟到物理地址的转换。这样的块称为一页。

实际上页表并不是一个由$2^{27}$个PTE直接组成的数组，而是由一个三层的树状索引组成。27位的索引被分成三个9位索引，对应的页表被分为三层每层由$2^9=512$个PTE组成。树的根是一个 4096 字节(4096/512=8,每个PTE有8字节即64位组成，其中44位PPN,10位Flags,还有10位暂时未被使用)的页表页，它包含 512 个 PTE，PTE中包含树下一层页表页的物理地址,根据下一层页表页物理地址找到下一个页表页后根据下一个9位索引在这层页表页中找到对应PTE，在最后一层的页表页中的PTE保存了指向虚拟地址映射的物理地址的PPN，将其和原始虚拟地址的12位Offset组合即可得到完整物理地址。

如果转换地址所需的三个 PTE 中的任何一个不存在，则分页硬件会引发页面错误异常。

  • PTE标志位

    ​ PTE_V 指示 PTE 是否存在：如果未设置，则对页面的引用会导致异常（即不允许）。 PTE_R 控制是否允许指令读取到页面。 PTE_W 控制是否允许指令写入页面。 PTE_X 控制 CPU 是否可以将页面内容解释为指令并执行它们。 PTE_U 控制是否允许用户态指令访问页面；如果未设置 PTE_U，则 PTE 只能在主管模式下使用。

  • satp寄存器

    非常非常重要的寄存器，保存着当前使用CPU的进程的根页表页的物理地址，是地址映射的起点，每个CPU都有它自己的satp寄存器，因此多个寄存器可运行多个进程。

3.2 内核地址空间

Xv6 为每个进程维护一个页表，描述每个进程的用户地址空间，除此之外，还维护一个描述内核地址空间的页表。

内核使用“直接映射”获取 RAM 和内存映射设备寄存器(QEMU 将设备接口作为位于物理地址空间中 0x80000000 以下的内存映射控制寄存器公开给软件)。除此之外，内核地址空间的某些虚拟地址不使用直接映射，像是trampoline页和内核堆栈页。为防止堆栈溢出覆盖掉其他内存，不同进程的内核堆栈页和trampoline页之间都有guard页，guard页的 PTE 无效（即 PTE_V 未设置），因此如果内核溢出内核堆栈，很可能会导致异常，内核会崩溃，而不是在用户不知情情况下错误运行。

3.3 Code: creating an address space

xv6中大多数用于操作地址空间和页表的代码在 vm.c 中。其中核心数据结构是 pagetable_t，它实际上是一个指向 RISC-V 根页表页的指针，pagetable_t 可以指向内核页表，也可以指向进程的页表。核心函数是 walk和 mappages。walk对给定的虚拟地址，在相应页表中查找对应的PTE；mappages则为新的映射装载PTE。以 kvm 开头的函数操作内核页表；以uvm开头的函数操作用户页表。copyout 和 copyin 向或者从用户地址空间拷贝数据，out和in是相对内核而言，copyout即将数据从内核地址空间拷贝到用户地址空间。

在boot阶段，main调用kvminit创建内核页表。此调用发生在 xv6 启用 RISC-V 上的分页之前，因此地址直接引用物理内存。kvminit 首先分配一个物理内存页来保存根页表页，然后调用kvmmap建立一系列直到PHYSTOP的直接映射。其中kvmmap调用mappages实现，mappages中第一个参数为内核页表时，即为kvmmap，为用户页表时也可用于为用户页表装载映射。由此时kvmmap中前两个参数都相同可见这一阶段进行的都是直接映射。

mappages函数在给定pagetable中建立从虚拟地址va到物理地址pa大小为size字节的标志位为perm的映射。

另一个核心函数是mappages中用到的walk函数。walk函数在给定的根页表页pagetable中查找虚拟地址va对应的最后一层PTE，并返回该PTE的地址。若查找到的PTE的PTE_V标志位为0，当alloc参数为0时，返回0表示查找失败；当alloc参数不为0时，建立一个新的空页表页，让查找到的PTE指向它并将PTE_V置为1，这个过程可能进行两次，建立起第二和第三级页表页。因此，理想情况下walk函数总是返回一个最后一层PTE的地址，如果存在新建过程，则该PTE的内容全为0(PTE_V也就为0)，否则查找到已经映射过的PTE情况下，PTE_V为1。

*main *调用 kvminithart来装载内核页表。它将根页表页的物理地址写入寄存器 satp。之后，CPU将使用内核页表进行地址转换。

main 调用 procinit 来为每个进程分配一个内核堆栈。它将每个堆栈映射到由 KSTACK 生成的虚拟地址，从而为无效的堆栈保护页面留出空间。 kvmmap 将映射 PTE 添加到内核页表，并且调用 kvminithart 将内核页表重新加载到 satp 以便硬件知道新的PTE。

为实现不同进程的内核堆栈之间添加保护页，宏函数KSTACK对进程的内核堆栈虚拟地址进行步长为2(个PGSIZE)的映射。

由于页表采用多级存储结构，在地址转换过程中需要多次访问内存进行查询，为避免由此带来的巨大开销，RISC-V CPU 在翻译后备缓冲区 (TLB) 中缓存页表条目，这样只有当TLB中不命中时，CPU才需要在多级页表中进行查询。使用新的页表后(比如进行进程切换)，需要刷新TLB，RISC-V 有指令 sfence.vma 刷新当前 CPU 的 TLB。 xv6在重新加载satp寄存器后在kvminithart中执行sfence.vma，在返回用户空间前切换到用户页表的trampoline代码中也执行sfence.vma。

3.4 Physical memory allocation & Code

xv6 使用内核末端和PHYSTOP 之间的物理内存进行运行时分配。它一次分配和释放整个 4096 字节的页面。

内存分配器的数据结构是一个可分配物理内存页的空闲链表。空闲列表的节点是一个run结构，每个可分配物理页对应的run结构储存在该页里。空闲链表和一个锁被包裹在一个结构中，以明确锁保护结构中的字段。锁的相关内容将在Chapter 6中说明。

函数main调用kinit初始化分配器，kinit 将内核末尾和 PHYSTOP 之间的每一页都初始化到空闲链表中。kinit 调用 freerange 将上述范围内的内存添加到空闲链表，而freeange通过对每页调用kfree实现。

由于PTE 只能指示在 4096 字节对齐的物理地址，所以freerange需要使用PGROUNDUP来保证释放的物理内存都是对齐的。分配器初始状态没有内存可用，通过使用kfree得到空闲内存。

kfree首先将待释放的内存的值全置为1来使得错误用到该内存的程序更快抛出错误而不是错误地运行。随后将该页添加到空闲链表中。具体添加方法是将pa转化为一个指向run的指针r，然后使用头插法将r插入到空闲链表中。

分配内存使用kalloc函数，kalloc从空闲链表的头部取下一个节点，将该页值全赋为5后返回指向这一物理页的指针。

3.5 Process address space

进程的用户内存从虚拟地址零开始，可以增长到 MAXVA ，原则上允许进程寻址 256 GB 的内存。当进程向xv6请求更多用户内存时，xv6首先用kalloc分配物理页，随后向进程的页表里添加指向这部分物理内存的PTE。

使用进程页表有诸多好处。不同的页表可以将用户地址转化成不同的物理页来加强不同进程之间的隔离；每个进程都可认为自己拥有从0开始的连续虚拟地址，尽管对应的物理地址可能并不连续；内核将有trampoline代码的同一个页映射到每个用户页的顶部，这样使得同一个物理页可以在多个地址空间中可见。

3.6 Code：sbrk

Sbrk 是进程缩小或增长内存的系统调用。该系统调用由函数growproc实现。 growproc 调用 uvmalloc 或 uvmdealloc，取决于参数 n 是正数还是负数。uvmalloc 使用 kalloc 分配物理内存，并使用 mappages 将 PTE 添加到用户页表中。 uvmdealloc 调用 uvmunmap，它使用 walk 来查找 PTE，并使用 kfree 来释放它们引用的物理内存。

uvmunmap使用walk每次查找一页待释放内存对应的PTE的地址，确认地址有效后使用kfree释放这一页物理内存，最后将对应这一页物理内存的PTE置为0使其恢复到unmap状态。在xv6中，用户进程的页表不仅告诉了分页硬件如何转换虚拟地址，也记录着哪些物理页已经被分配给用户进程，因此在回收分配给用户的物理页之前，应该先检查相关的PTE是否存在/有效。

3.7 Code：exec

Exec 是创建地址空间的用户部分的系统调用。它从存储在文件系统中的文件初始化地址空间的用户部分。从指定路径打开文件后，exec首先检查文件的elf头。ELF二进制文件以“magic number”0x7F、“E”、“L”、“F”这四个字节或 ELF_MAGIC 开头，exec检查到这样的开头后，即认为该文件是一个可用的elf二进制文件。

Exec 使用 proc_pagetable 分配一个没有用户地址映射的新页表，分配的新页表中只有trampoline和tramframe的映射。之后使用uvmalloc为每个ELF段分配内存。使用loadseg调用walkaddr将ELF段装入前面分配的内存中。将ELF段从文件中读取的工作则由readi完成。之后exec在页表中分配并初始化用户栈。

在allocproc中添加对进程内核页的初始化操作，初始化的函数ukpinit通过参考kvminit实现。具体是在函数中新创建一个页表，然后通过使用mappages向新建的页表而不是kvmmap来向固定的单一内核页表添加映射，最后返回已经添加和原内核页表相同的映射的新页表。

除了要在进程内核页中添加和kvminit中向内核页表添加的映射外，还需要创建当前进程的内核栈并向进程内核页中添加当前进程的内核栈的映射。原来的全局内核页表方案中，所有进程的内核栈都在同一个内核页表中的不同位置映射，在procinti函数中统一进行，而在新方案中，只需要在每个进程的进程内核页中加入当前进程的内核栈的映射即可。此处可选择像原方案一样对不同的进程选择不同的内核栈映射存放位置，也可改为将该映射存放到固定位置。

在proc.c的调度程序scheduler函数中添加切换页表的代码。在进程运行前切换到进程内核页，并在进程结束运行后恢复使用全局内核页表。

最后，由于对在进程控制结构中加入了进程内核页且申请了进程的内核栈，因此在释放进程相关资源时也要相应释放掉进程内核页和内核栈的资源。

因为walkaddr只能在用户空间搜索物理地址，因此只能用walk先找到存放kstack的pte项的地址，然后用PTE2PA从pte中提取出kstack的物理地址进行释放。

由于不同进程的进程内核页实际上映射到的是相同的物理地址，因此在释放进程内核页时只能释放页表而不能释放最后一级页表中pte指向的实际物理空间，参考freewalk函数实现一个释放一二三级页表但是不释放最后一级的页表中pte指向的物理空间。
